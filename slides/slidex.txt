2) Historical context

I started working on goany as I've many times during my career that we needed to implement
same thing few times in different languages. Perfect, would be to have just one language
and then compile it/transpile it to other languages in deterministic way.

There are some solutions now. First of all we write it once in native language and expose it via
FFI API. There are also solutions similar to goany but they are based on concept of independent
programming language for instance fusionlang and haxe. Both are independent languages that can be
transpiled into several backends like C/C++, Rust, Java and so on.

4) Goany base principles and it's evolution

I've chosen different way than previously described projects and approaches.
I've chosen go programming language instead of designing new programming language
with the goal to compile it / transpile it to other backends like C++/Rust/C#/Java.

There are few reasons for that, tradeoffs and challenges. First if you design new programming
language you have total freedom about syntax and semantics. You have to just design mapping
to all supported backends which still is challenging.

If you use existing language, there are questions about what features we would like support.
Supporting whole language might be not realistic or hard to do, however seems feasible.

What kind of optimizations goany wants to support (research)
The big performance wins in real programs come from:

  - Memory layout — cache misses cost 100+ cycles, an instruction costs 1
  - Data movement — CPU↔GPU, RAM↔cache, core↔core
  - Algorithmic access patterns — sequential vs random
  - Parallelism — using all cores/threads

  Instruction-level optimization — register allocation, instruction scheduling, peephole — saves single-digit percentages. Important, but incremental.

  Your SoA transformation gave 3x on C++. No amount of instruction-level optimization would do that. LLVM was already doing its best on the AoS code — the bottleneck was cache misses, which no instruction reordering can fix.

  The irony is that production compilers have spent decades perfecting the micro level while ignoring the macro level. They'll spend enormous effort to save one register spill but won't touch the data layout that causes millions of
  cache misses.

  Pointer lowering operates at the macro level — it changes how data is organized, not how instructions execute. That's where the 2x-10x wins live. And it's essentially uncontested territory.
